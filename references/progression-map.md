# Progression Map — what "next" looks like from each cell

Use this after placement, once the person has told you their goals. The map offers directions, not obligations: moving deeper in a column and stepping into a new column are equally valid. Help them pick **one** cell to grow into and **one** next step startable this week.

Two principles carried through every suggestion:

- **Critical experiences, not content.** People advance by doing specific things, not by watching courses. Each move below names the experiences that actually cause the transition.
- **Unequal step sizes.** The moves are not the same size. The big leaps are marked ⚠ — set expectations honestly when someone faces one.

## Augmentation moves

**→ into Foundations** (from little/no use)
Pick three recurring tasks from your actual week and run each through an AI tool five times. The goal is calibration — learning what it's good at — not perfection. Critical experiences: 15–20 completed real-work tasks; noticing the first task where AI is now your default.

**Foundations → Essentials**
Stop accepting first drafts. Learn structured prompting with context control: give the model your source material, an example of good output, and explicit constraints. Critical experiences: ship an AI-assisted artifact a colleague or client actually uses; build a personal prompt library with 5+ entries you return to; teach one technique to one colleague.

**Essentials → Fluency**
Move from prompting to building: create one custom assistant (custom GPT, Gem, Claude skill/project) that encodes your context permanently so you stop re-explaining. Critical experiences: the first build; the first *rebuild* after real use teaches you what was wrong; integrating it into your daily stack.

**Fluency → Native** ⚠
This is an operating-system change, not a feature addition. Redesign your workflow so delegation is the default: start each day by asking what can be handed off. Critical experiences: a full week where every recurring task got triaged for delegation; discovering which judgment calls genuinely can't be delegated (that's the actual lesson).

## Automation moves

**→ into Foundations** (often from Augmentation Essentials+)
Map one recurring team process end-to-end: every step, branch, and failure mode. Don't build yet — the mapping skill is the foundation. Critical experiences: one honest process map including error paths; identifying which steps need judgment vs. which follow rules.

**Foundations → Essentials**
Build the thing you mapped, on a no-code platform, taking vibecoding seriously as a construction method. Critical experiences: first working interactive tool; first user who isn't you; first fix after a user breaks it in a way you didn't predict.

**Essentials → Fluency**
Chain steps into multi-step workflows with agentic nodes and platform integrations. Critical experiences: a workflow with 3+ steps and a real failure-handling path; documenting it well enough that a colleague runs it without you; measuring the time it actually saves.

**Fluency → Native** ⚠
Production discipline: versioning, monitoring, handoff protocols. The skill shifts from "building things that work" to "building things that keep working when you're not looking." Critical experiences: first versioned rollback; writing a handoff doc and actually handing off; a month of someone else operating your system.

## Agentic moves

**→ into Foundations** (usually from Automation Fluency+, or technically curious Augmentation Fluency)
Learn agentic design principles by architecting one AI-native solution on paper: what context does the agent need, what should it remember, what tools does it get, where must a human stay in the loop. Critical experiences: one written architecture; explaining context management to someone else and surviving their questions.

**Foundations → Essentials**
Build an organizational knowledge system: MCP integrations, custom skills, decision logic documented so the system's behavior is inspectable. Critical experiences: first MCP integration or custom skill; writing decision logic down and watching the system follow it; first correction cycle when it doesn't.

**Essentials → Fluency**
Adopt PRD-driven development: spec before build, testing protocols, deployment that isn't "it runs on my machine." Critical experiences: a PRD someone else reviewed; a test suite that caught a regression; shipping to users who never met you.

**Fluency → Native** ⚠ (the biggest leap on the map)
Scale across teams: governance frameworks, organizational memory architecture, and the political-organizational work that engineering alone can't do. Critical experiences: a governance decision that constrained your own system; onboarding a second team; designing memory/knowledge architecture that outlives any single project. Expect quarters, not weeks.

## Choosing between "deeper" and "wider"

When goals don't obviously decide it:

- **Go deeper** in the current column when the person's role rewards mastery there (an analyst deepening Augmentation, an ops lead deepening Automation).
- **Go wider** into the next column when their impact is capped by radius — they're personally fast but their team is slow (→ Automation), or their automations are multiplying and colliding (→ Agentic).
- When torn, wider-at-Foundations usually beats deeper-at-Native: the ⚠ leaps are expensive, and first-column-to-the-right foundations often return more value per effort.
